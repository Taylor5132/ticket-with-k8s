## ① 시스템 개요

본 시스템은 KOPIS 공연 카탈로그를 기반으로 한 공연 예매 플랫폼으로, 5개의 백엔드 서비스와 1개의 프론트엔드로 구성된 마이크로서비스 아키텍처다. Redis 기반 대기열(Waiting Queue)과 비동기 스트림으로 예매 트래픽 유입량을 제어하는 것이 핵심 특징이다.

## ② 서비스별 기능 목록

### auth-service (사용자 인증/계정 관리)

| 기능명 | 설명 |
|---|---|
| 로컬 회원가입 | `POST /auth/signup`. login_id(4~20자 영문·숫자·밑줄 정규식), password(8자 이상), display_name(공백 아님) 검증 후 bcrypt 해시. provider='local'로 users INSERT, 중복 시 409 ID_TAKEN. 성공 시 JWT 발급. |
| 로컬 로그인 | `POST /auth/login`. provider='local' 사용자의 password_hash를 bcrypt.checkpw로 검증. 실패 시 401 INVALID_CREDENTIALS, 성공 시 JWT 발급. |
| 개발용 로그인(dev-login) | `POST /auth/dev-login`. 임의 provider/login_id/display_name으로 사용자 upsert(ON CONFLICT(provider,login_id) DO UPDATE) 후 JWT 발급. 비밀번호 검증 없음(개발/테스트용). |
| Google OAuth 로그인 시작 | `GET /auth/google`. CSRF용 state 토큰(secrets.token_urlsafe) 생성, httponly oauth_state 쿠키(max_age 300, samesite=lax) 설정 후 Google authorization URL로 302 리다이렉트. GOOGLE_CLIENT_ID 미설정 시 503. |
| Google OAuth 콜백 | `GET /auth/google/callback`. oauth_state 쿠키와 state 파라미터 일치 검증, Google token/userinfo API 호출, provider='google'로 사용자 upsert, JWT 발급 후 FRONTEND_URL로 토큰·user를 쿼리스트링에 담아 리다이렉트. 오류 시 oauth_error 파라미터로 리다이렉트. |
| 내 정보 조회 | `GET /auth/me`. JWT(current_user) 인증 후 토큰 sub(id)로 users에서 사용자 정보 조회. |
| JWT 발급/검증 | issue_token: HS256, JWT_SECRET 서명, payload(sub, provider, display_name, iat, exp), 만료 12시간. current_user 의존성: Authorization Bearer 헤더 파싱·디코드, 실패 시 401 UNAUTHORIZED. |
| 사용자 upsert 유틸 | upsert_user: users에 INSERT ... ON CONFLICT(provider, login_id) DO UPDATE SET display_name, updated_at=now(). make_user_id로 login_id 기반 안전한 user id 생성. |
| 스키마 자동 마이그레이션 | startup 이벤트에서 ALTER TABLE users ADD COLUMN IF NOT EXISTS password_hash TEXT 실행. |
| 관측성/모니터링 | telemetry.configure_tracing로 OTEL 트레이싱(FastAPI/SQLAlchemy/HTTPX, OTLP 엔드포인트 설정 시), prometheus_fastapi_instrumentator로 /metrics 노출. CORS 전체 허용. |
| 헬스체크 | `GET /health` → {status: ok}. |

| 기능명 | 설명 |
|---|---|
| 공연 목록 조회/필터링 | performances를 venues와 조인해 공연 카드 목록 반환. genre, area(province), status 쿼리 파라미터로 필터링, start_date·id 순 정렬. |
| 공연 상세 조회 | 공연 ID로 공연+공연장 상세(포스터, 장르, 상태, 기간, 출연진, 러닝타임, 관람등급, 소개이미지, 공연장 정보 등) 반환. is_open_run은 'Y' 여부로 bool 변환, 위도/경도 float 변환. |
| 공연 회차(스케줄) 계산 | compute_schedules/_schedule_days로 KOPIS 스타일 한글 요일 텍스트를 파싱해 start_date~end_date 사이 해당 요일 날짜 목록(최대 20개) 생성. 파싱 불가/공백 시 주말{토,일}로 폴백. |
| 고정 가격표 제공 | price_text()로 'VIP석 150,000원, R석 120,000원, S석 90,000원, A석 60,000원' 문자열을 상세 응답에 포함. |
| 좌석 등급/가격 정의 (내부) | seat_id 접두(행 문자 A~H)를 GRADE_RULES에 매핑해 grade·price 계산. DB 미접근 순수 계산. A/B→VIP 150000, C/D→R 120000, E/F→S 90000, G/H→A 60000. |
| 소개 이미지 파싱 | list_intro_images로 '|' 구분 raw 문자열을 이미지 URL 리스트로 변환(빈 항목 제외). |
| 헬스체크 | `/health` → {status: ok}. |
| 관측성(Observability) | OTLP 엔드포인트 설정 시 OpenTelemetry FastAPI/SQLAlchemy 트레이싱, prometheus-fastapi-instrumentator로 /metrics 노출, CORS 전체 허용. |

### booking-service (예매/대기열)

| 기능명 | 설명 |
|---|---|
| 좌석 가용성 조회 | 공연/일자별 A-H행 1-10번 총 80석 생성, bookings에 예매된 좌석을 OCCUPIED로 표시해 반환. 등급/가격은 행 기준 고정(GRADE_RULES). |
| 예매 요청 생성(비동기) | JWT 사용자가 좌석 예매 요청 시 booking_requests에 PENDING 기록 후 'booking.requests' 스트림에 XADD, request_id 반환. 실제 처리는 워커가 비동기 수행. ENFORCE_ADMISSION_TOKEN=true면 입장 토큰 없이는 403 차단. |
| 예매 요청 상태 조회 | request_id로 본인 요청 상태(PENDING/PROCESSING/CONFIRMED/FAILED), 실패 사유, 확정 booking_id, show_date 폴링 조회. |
| 대기열 진입(Waiting Queue join) | 공연/일자 대기줄 진입. 처음 진입자만 INCR로 번호표를 받아 ZSET에 ZADD(nx)하고 active-queues에 등록 → 선착순 FIFO. 본인 position·전체 total 반환. 이미 입장 토큰 있으면 즉시 admitted. |
| 대기열 상태 조회 | 입장 토큰 존재 여부로 admitted 판정. 토큰 있으면 admitted, 줄에 있으면 position/total, 줄에서 빠졌으나 토큰 미발급(스트림 처리 중)이면 대기 응답. |
| 내 예매 목록 조회 | JWT 사용자의 확정 예매 내역(공연명/공연장/일자/좌석/등급/결제액/예매시각)을 booked_at 내림차순 반환. |
| [Worker] 예매 비동기 처리 | worker.py: 'booking.requests'를 Consumer Group으로 소비. pg_advisory_xact_lock(좌석 단위 락)으로 동시성 제어 후 중복 확인 → event-service 좌석 조회 → payment-service에 X-Service-Token 포인트 차감 → bookings INSERT(ON CONFLICT DO NOTHING) → CONFIRMED/FAILED 갱신. 처리 후 XACK. |
| [Dispatcher] 대기열 유입량 제어 | dispatcher.py: TICK_SECONDS(기본 1초)마다 active-queues 각 큐에서 (ADMISSION_RATE×TICK, 기본 3/s)명을 Lua(DISPATCH_LUA: ZPOPMIN+XADD 원자)로 work-stream에 핸드오프. 빈 큐는 active-queues에서 SREM. |
| [Admission Worker] 입장 토큰 발급 | admission_worker.py: work-stream을 'admission-workers' 그룹으로 소비해 입장 토큰(token:..., TTL=ADMISSION_TOKEN_TTL) 발급 후 XACK. 주기적 XAUTOCLAIM으로 죽은 워커의 idle(기본 30s) 일감 회수·재처리 → 무유실 보장. KEDA가 work-stream lag 기준 스케일. |
| [공통] 멱등 Consumer Group 생성 | FastAPI startup 시 ensure_admission_group으로 work-stream의 admission-workers 그룹 MKSTREAM+멱등 생성(Redis 미가용 시 기동 막지 않음). worker는 ensure_group으로 booking.requests의 booking-workers 그룹 멱등 생성. |
| [운영] 분산 트레이싱/메트릭 | telemetry.configure_tracing이 OTLP 엔드포인트 설정 시 OTLP gRPC로 FastAPI/SQLAlchemy/Redis 트레이싱 활성화. prometheus_fastapi_instrumentator로 /metrics 노출. |

### payment-service (포인트 결제)

| 기능명 | 설명 |
|---|---|
| 포인트 잔액 조회 | JWT 인증 사용자의 포인트 잔액 조회. 레코드 없으면 user_id에 'demo-rich' 포함 시 300000, 아니면 100000으로 자동 생성(ensure_balance) 후 반환. |
| 결제 내역 조회 | JWT 인증 사용자의 payment_history를 paid_at 내림차순 조회해 booking 정보, 공연명, 금액, 상태 등 반환. |
| 포인트 차감(결제 처리) | X-Service-Token으로 인증된 내부 호출로 예매 건 포인트 차감. booking_request_id 단위 멱등 처리(존재 시 기존 결제 재반환), point_balances 행 잠금(FOR UPDATE) 후 잔액 검증·차감, payment_history에 'PAID' 기록. |
| 포인트 잔액 자동 초기화 | ensure_balance: point_balances에 행 없을 때 기본 잔액(demo-rich 300000, 그 외 100000)으로 INSERT하여 잔액 보장. |
| 멱등 결제 처리 | deduct에서 booking_request_id로 기존 결제 존재 여부 확인해 중복 차감 방지(존재 시 기존 payment_id·현재 잔액 반환). |
| JWT 인증 | Authorization: Bearer 토큰을 JWT_SECRET/HS256으로 디코드해 sub, display_name, provider 추출하는 current_user 의존성. 실패 시 401 UNAUTHORIZED. |
| 서비스 간 인증 | X-Service-Token 헤더를 SERVICE_TOKEN과 비교해 내부 차감 API 보호. 불일치 시 403 FORBIDDEN. |
| 헬스체크 | `GET /health` → {status: ok}. |
| 관측성(Telemetry) | OTLP 엔드포인트 설정 시 OpenTelemetry 트레이싱(FastAPI+SQLAlchemy), prometheus-fastapi-instrumentator로 /metrics 노출, CORS 전체 허용. |

### saved-service (공연 찜)

| 기능명 | 설명 |
|---|---|
| 찜한 공연 목록 조회 | 로그인 사용자 Redis Set의 공연 ID들을 (숫자면 정수 기준) 정렬 후, 각 ID에 대해 event-service /performances/{id} 호출해 카드(id, title, poster_url, venue_name, area, genre, start_date, end_date)로 변환해 items 반환. 200이 아닌 ID는 제외. |
| 공연 찜 추가 | 로그인 사용자 Redis Set(saved:user:{user_id})에 performance_id를 sadd로 추가, saved:true 반환. |
| 공연 찜 삭제 | 로그인 사용자 Redis Set에서 performance_id를 srem으로 제거, saved:false 반환. |
| JWT 인증 | current_user 의존성으로 Bearer 토큰 검증. JWT_SECRET/HS256 디코드 후 payload['sub']를 사용자 id로 사용. 누락/디코드 실패 시 401 UNAUTHORIZED. |
| 헬스 체크 | `GET /health` → {status: ok} (인증 불필요). |
| 공연 카드 변환 | card() 헬퍼가 event-service 상세 응답을 목록용 카드 스키마로 변환. venue.name → venue_name, venue.province → area 매핑. |
| 관측성(Observability) | OpenTelemetry 분산 트레이싱(FastAPI/Redis/HTTPX, OTLP 엔드포인트 설정 시) 및 prometheus-fastapi-instrumentator로 /metrics 노출. CORS 전체 허용. |

### 페이지별 제공 기능

#### 공연 목록 (대시보드/홈) — `/`

- 전체 공연 목록 조회 및 포스터 그리드 표시
- 상단 배너 노출 (공연 목록 기반)
- 장르별 필터 (목록에서 동적 집계, 최대 12개, 토글 선택)
- 지역별 필터 (목록에서 동적 집계, 최대 12개, 토글 선택)
- 필터 초기화 버튼 (장르·지역 동시 해제)
- '오픈 예정' 섹션: 공연예정 상태이며 D-1~D-3 이내 공연을 D-day 뱃지와 함께 정렬 노출
- 활성 필터에 따라 섹션 제목과 공연 수(count) 갱신
- 로딩 중 안내 문구 표시
- 공연 카드 클릭 시 상세 페이지로 이동

#### 로그인 / 회원가입 — `/login`

- 로그인/회원가입 탭 전환
- 아이디·비밀번호 기반 로컬 로그인 (클라이언트 입력 검증)
- 회원가입 (아이디 4~20자 영문·숫자·밑줄, 비밀번호 8자 이상, 비밀번호 확인 일치, 닉네임 입력 검증, 가입 시 10만 포인트 안내)
- Google로 시작하기 버튼 (OAuth 활성 시 /api/auth/google 리다이렉트, 비활성 시 dev-login 폴백)
- 데모 계정(demo-basic 100,000P / demo-rich 300,000P) 원클릭 체험 로그인
- 로그인/가입 성공 시 직전 경로(from)로 복귀 또는 홈으로 이동
- 에러 메시지 및 제출 중 버튼 비활성화 처리
- 이미 로그인된 사용자는 자동 리다이렉트

#### 공연 상세 — `/performances/:id`

- 공연 상세 정보 조회 (포스터, 장르, 제목, 공연장/지역, 공연 기간, 러닝타임, 관람 연령)
- 관람 가능 날짜(스케줄) 선택 (날짜 칩 토글)
- 관심공연 저장/해제 토글 (비로그인 시 로그인 페이지로 유도)
- 현재 공연의 관심공연 저장 여부 표시 (로그인 시 saved/me로 확인)
- '예매하기' 버튼: 선택한 날짜를 쿼리로 좌석 선택 페이지 이동 (날짜 미선택 시 비활성)
- 탭 전환: 상세정보(출연/소개 이미지), 좌석/가격(등급별 가격표 + 공식 가격 안내), 관람안내
- 로딩 중 안내 문구 표시

#### 좌석 선택 / 예매 — `/performances/:id/seats?show_date=...`

- 대기열(큐) 진입 및 입장 허가 대기 (admitted=false 시 대기 화면, 내 대기순서·진행바 표시)
- 대기열 상태 2초 간격 폴링하여 입장 허가 시 자동 진입
- 좌석 배치도 렌더링 (행별 그룹화, STAGE 표시, 등급별 색상)
- 좌석 선택/해제 토글 (예매 완료 좌석은 비활성)
- 좌석 등급 범례 표시 (VIP/R/S/A 가격, 예매 완료)
- 선택 좌석 목록 및 총 좌석 수·합계 금액 계산
- '결제하기'로 선택한 좌석 각각 예매 요청 생성 후 예매 진행 페이지로 이동
- 비로그인 상태에서 결제 시도 시 로그인 페이지로 유도
- 입장 토큰 만료(NO_ADMISSION_TOKEN) 시 선택 초기화 후 대기열 재진입
- 날짜 미선택 시 상세 페이지로 안내
- 대기열 확인 중/좌석 로딩 중 안내 문구 표시

#### 예매 진행 상태 — `/booking?ids=...`

- 쿼리의 예매 요청 ID 목록에 대한 처리 상태 1초 간격 폴링
- 모든 요청이 CONFIRMED 또는 FAILED 될 때까지 처리 중 화면(스피너) 표시
- 전체 성공 시 예매 완료 화면 (공연 날짜, 마이페이지/공연 목록 이동 버튼)
- 전체/일부 실패 시 실패 화면 및 사유별 안내 메시지 (SEAT_ALREADY_BOOKED, INSUFFICIENT_POINTS, PAYMENT_FAILED, WORKER_ERROR)
- 일부 성공 시 성공 좌석 수 안내

#### 마이페이지 — `/mypage`

- 프로필 표시 (닉네임, 로그인 제공자 라벨, 보유 포인트)
- 보유 포인트 잔액 조회
- 예매내역 목록 (공연 제목, 공연장, 좌석/등급/결제금액, 공연일, 예매일)
- 최근 결제내역 목록 (공연 제목, 결제 금액, 결제일)
- 관심공연 그리드 표시
- 비로그인 시 로그인 유도 화면 (from=/mypage)
- 로딩 중 안내 문구 표시

#### 전역 헤더/푸터 (모든 페이지 공통 레이아웃) — (App.tsx layout)

- 브랜드 로고 클릭 시 홈 이동
- '공연' 내비게이션 링크
- 로그인 상태: 사용자 이름 표시, 마이페이지 아바타 링크, 로그아웃 버튼 (로컬 세션 정리)
- 비로그인 상태: 로그인 링크
- 푸터: 프로젝트 데모 안내 및 공연 정보 출처(KOPIS) 표기

### 예매 흐름 (Booking Flow)

```
[홈 /]  공연 목록 조회 (GET /api/performances)
   │ 카드 클릭
   ▼
[공연 상세 /performances/:id]  상세 조회 (GET /api/performances/:id)
   │ 날짜(스케줄) 선택 후 '예매하기'
   ▼
[좌석 선택 /performances/:id/seats?show_date=...]
   │ 1) 대기열 진입 (POST /api/queue/join) → admitted 여부 확인
   │ 2) (대기 시) GET /api/queue/status 2초 폴링 → admitted=true 시 진입
   │ 3) 좌석 가용성 조회 (GET .../seat-availability?show_date=...)
   │ 4) 좌석 선택 후 '결제하기' → 좌석마다 POST /api/booking-requests
   │    · 비로그인 시 로그인 페이지 유도
   │    · NO_ADMISSION_TOKEN 시 선택 초기화 후 대기열 재진입
   ▼
[예매 진행 /booking?ids=...]
   │ GET /api/booking-requests/:id 1초 폴링
   │ 모든 요청 CONFIRMED/FAILED 까지 처리 중 화면
   ▼
 전체 성공 → 예매 완료 화면 / 실패 → 사유별 안내
   │
   ▼
[마이페이지 /mypage]  예매내역·결제내역·포인트 잔액 확인
```

### 대기열 흐름 (Waiting Queue Flow) — 프론트 관점

```
좌석 선택 페이지 진입
   │
   ▼
POST /api/queue/join?performance_id=...&show_date=...
   │
   ├─ admitted=true (입장 토큰 보유) ──────────► 좌석 선택 화면 즉시 진입
   │
   └─ admitted=false ──► 대기 화면 표시 (내 position / total, 진행바)
                              │
                              ▼ 2초 간격
                         GET /api/queue/status?... 폴링
                              │
                              ├─ admitted=true ──► 좌석 선택 화면 자동 진입
                              └─ admitted=false ─► 대기 화면 유지 (위치 갱신)
```

> 대기열의 입장 허가는 백엔드의 dispatcher / admission-worker 파이프라인이 비동기로 부여한다(아래 ④ 참조). 프론트는 토큰 발급 결과를 폴링으로 감지한다.

## ④ 백그라운드 / 시스템 기능

### 비동기 컴포넌트 요약

| 컴포넌트 | 파일 | 소비 스트림 / Consumer Group | 주요 역할 |
|---|---|---|---|
| booking-worker | worker.py | `booking.requests` / `booking-workers` | 예매 요청 비동기 처리·확정 |
| dispatcher | dispatcher.py | (스트림 소비 아님, ZSET→스트림 핸드오프) | 대기열 유입량 제어 |
| admission-worker | admission_worker.py | `work-stream` / `admission-workers` | 입장 토큰 발급, 죽은 워커 일감 회수 |

### booking-worker — 예매 확정

`booking.requests` 스트림을 `booking-workers` Consumer Group으로 소비하여 예매를 실제로 확정한다.

처리 순서:

| 단계 | 동작 |
|---|---|
| 1 | 스트림에서 예매 요청 수신 (XREADGROUP) |
| 2 | pg_advisory_xact_lock으로 좌석 단위 락 획득 (동시성 제어) |
| 3 | 좌석 중복 예매 여부 확인 |
| 4 | event-service에서 공연/좌석(등급·가격) 정보 조회 |
| 5 | payment-service `/payments/deduct`에 X-Service-Token으로 포인트 차감 요청 |
| 6 | bookings INSERT (ON CONFLICT DO NOTHING으로 좌석 유니크 보장) |
| 7 | booking_requests 상태를 CONFIRMED 또는 FAILED로 갱신 |
| 8 | 처리 완료 후 XACK |

실패 사유(failure_reason): `SEAT_ALREADY_BOOKED`, `INSUFFICIENT_POINTS`, `PAYMENT_FAILED`, `WORKER_ERROR`.

### dispatcher / admission-worker — 대기열

대기열은 두 컴포넌트가 협력한다. dispatcher가 줄에서 사용자를 뽑아 work-stream으로 넘기고, admission-worker가 입장 토큰을 발급한다.

#### dispatcher (유입량 제어)

| 항목 | 내용 |
|---|---|
| 주기 | TICK_SECONDS 마다 실행 (기본 1초) |
| 대상 | active-queues Set에 등록된 각 큐 |
| 처리량 | 큐당 (ADMISSION_RATE × TICK)명, 기본 3/s |
| 핸드오프 방식 | Lua 스크립트 DISPATCH_LUA로 ZPOPMIN + XADD를 원자 실행 → work-stream으로 전달 |
| 정리 | 빈 큐는 active-queues에서 SREM |
| 효과 | 줄에서 빼는 속도로 결제/예매 시스템 유입량 조절 |

#### admission-worker (입장 토큰 발급)

| 항목 | 내용 |
|---|---|
| 소비 | work-stream을 admission-workers Consumer Group으로 소비 |
| 발급 | 사용자별 입장 토큰 `token:{perf}:{date}:{user_id}` (SET ex=ADMISSION_TOKEN_TTL, 기본 600초) |
| 완료 | 토큰 발급 후 XACK |
| 무유실 보장 | 주기적(CLAIM_EVERY 루프마다) XAUTOCLAIM으로 죽은 워커가 PEL에 남긴 idle(기본 30s) 일감을 회수·재처리 |
| 스케일링 | KEDA가 work-stream lag 기준으로 스케일 |

### 비동기 파이프라인 (전체 흐름)

```
[사용자] queue/join
   │ INCR seq:{perf}:{date} → 번호표(score)
   ▼
ZSET queue:{perf}:{date}  (FIFO 대기열, ZADD nx)
   │  + active-queues Set 등록
   ▼
[dispatcher]  TICK_SECONDS마다 (ADMISSION_RATE×TICK)명
   │  Lua: ZPOPMIN + XADD (원자 핸드오프)
   ▼
work-stream  (Consumer Group: admission-workers)
   │
   ▼
[admission-worker]  XREADGROUP → 입장 토큰 발급(token:..., TTL) → XACK
   │  (죽은 워커 일감은 XAUTOCLAIM으로 회수·재처리)
   ▼
사용자: GET /queue/status에서 토큰 존재 확인 → admitted=true
   │
   ▼
[사용자] POST /booking-requests (ENFORCE_ADMISSION_TOKEN=true 시 토큰 필수)
   │  booking_requests PENDING 기록 + XADD
   ▼
booking.requests 스트림  (Consumer Group: booking-workers)
   │
   ▼
[booking-worker]  좌석 락 → 중복확인 → event-service 조회
   │              → payment-service 포인트 차감 → bookings INSERT
   │              → CONFIRMED/FAILED 갱신 → XACK
   ▼
사용자: GET /booking-requests/:id 폴링으로 결과 확인
```

### 관련 Redis 키 / TTL

| 키 / 구조 | 용도 | TTL |
|---|---|---|
| `booking.requests` (STREAM) | 예매 요청 비동기 큐 (XADD/XREADGROUP, Consumer Group booking-workers) | - |
| `queue:{perf}:{date}` (ZSET) | 대기열 FIFO (번호표 score로 ZADD/ZRANK/ZCARD) | QUEUE_TTL 기본 3600초 |
| `seq:{perf}:{date}` (INCR) | 원자적 번호표 발급 | QUEUE_TTL 기본 3600초 |
| `active-queues` (Set) | Dispatcher 순회 대상 큐 레지스트리 | - |
| `work-stream` (STREAM) | 입장 처리 스트림 (Consumer Group admission-workers, Lua ZPOPMIN+XADD 핸드오프) | - |
| `token:{perf}:{date}:{user_id}` (SET) | 입장 토큰 | ADMISSION_TOKEN_TTL 기본 600초 |
| `saved:user:{user_id}` (Set) | 사용자별 공연 찜 목록 (saved-service) | - |

### 공통(부트스트랩) 기능

| 기능 | 설명 |
|---|---|
| 멱등 Consumer Group 생성 | FastAPI startup 시 ensure_admission_group으로 work-stream의 admission-workers 그룹을 MKSTREAM+멱등 생성(Redis 미가용 시 기동 막지 않음). worker는 ensure_group으로 booking.requests의 booking-workers 그룹 멱등 생성. |
| 서비스 간 인증 | booking-worker → payment-service `/payments/deduct` 호출 시 X-Service-Token 헤더 사용. payment-service는 SERVICE_TOKEN과 비교, 불일치 시 403 FORBIDDEN. |
| 멱등 결제 | payment-service deduct는 booking_request_id 단위로 멱등 처리(존재 시 기존 payment_id·현재 잔액 반환)하여 중복 차감 방지. |
